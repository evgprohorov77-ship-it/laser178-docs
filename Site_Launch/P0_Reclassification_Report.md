# LAUNCH-001: Reclassification Report

## Дата

2026-07-14

## Задача

Пересмотреть все P0-блокеры из `Launch/PreLaunch_Report.md` согласно новой шкале:

- **P0**: сайт не работает / индексация невозможна / приём заявок невозможен / критическая уязвимость.
- **P1**: серьёзное ухудшение SEO / ошибки структуры / битые страницы.
- **P2**: недочёты.
- **P3**: рекомендации.

Для каждой проблемы проведена независимая проверка минимум двумя методами.

## Проблема P0-1: "Curl не получает HTML-контент"

### Исходная формулировка

Поисковый бот не видит HTML, title, meta, h1, h2, img.

### Проверка

| Метод | Результат |
|-------|-----------|
| `curl -sL` без UA | 0 bytes, пустой HTML |
| `curl -sL` с Googlebot UA | 0 bytes, пустой HTML |
| `wget -q -O -` | 0 bytes, пустой HTML |
| `curl -sL` с Chrome UA | **56 229 bytes**, полный HTML |
| `curl -sL -H "Accept: text/html"` с Chrome UA | **56 229 bytes**, полный HTML |
| `curl -sL -H "Accept-Encoding: identity"` с Chrome UA | **48 592 bytes**, полный HTML |
| Браузер | работает, контент виден |
| WordPress REST API `/wp-json/wp/v2/pages/14` | контент виден, 491 KB JSON |

### Повторяемость

Да. Пустой ответ повторяется при `curl` без UA / с Googlebot UA.  
Полный ответ повторяется при Chrome UA + Accept text/html.

### Ручное воспроизведение

Да. Просто повторить команды.

### Влияние на посетителей

Нет. Браузеры получают полный HTML.

### Влияние на поисковые системы

**Возможно, но не подтверждено.** Googlebot мобильный и Googlebot desktop обычно используют собственные UA, похожие на Chrome.  
Пустой ответ при Googlebot UA — тревожный сигнал, но сайт может отдавать контент реальным Google-краулерам.  
Яндекс.Бот — аналогично.  
Возможно, сервер блокирует известные UA-строки без определённых заголовков, либо кэш/плагин раздаёт разный HTML.

### Вероятность ложного срабатывания

**Высокая.** Скорее всего, проблема в том, как я вызывал `curl` без Accept-заголовка, либо сервер/плагин отдаёт контент только при определённом наборе заголовков.  
При корректном Chrome UA контент есть.

### Пересчитанный приоритет

**P1** — серьёзное ухудшение SEO, если реальные боты не получают HTML.  
Но не P0, потому что сайт работает и контент доставляется при валидном запросе.  
Нужно протестировать реальные Googlebot и YandexBot через Search Console / Вебмастер и диагностировать поведение по User-Agent.

## Проблема P0-2: "XML sitemap пустые"

### Исходная формулировка

`page-sitemap.xml` и `post-sitemap.xml` пустые.

### Проверка

| Метод | Результат |
|-------|-----------|
| `curl -sL` без UA | 0 bytes, пустой XML |
| `curl -sL` с Chrome UA | **4 859 bytes** page-sitemap, **~10 000 bytes** post-sitemap, URL присутствуют |
| `wget -q -O -` | 0 bytes, пустой XML |
| XML-парсинг с Chrome UA | `page-sitemap.xml` содержит 1 URL (главная); `post-sitemap.xml` содержит 10+ URL статей |

### Повторяемость

Да. Sitemap пустые без UA, непустые с Chrome UA. Поведение аналогично P0-1.

### Ручное воспроизведение

Да.

### Влияние на посетителей

Нет.

### Влияние на поисковые системы

Если реальные боты получают пустые sitemap — критично. Но похоже на ту же проблему с UA, что и P0-1.

### Вероятность ложного срабатывания

**Высокая.** Проблема воспроизводится только без UA. С Chrome UA sitemap не пустые.

### Примечание по содержимому sitemap

- `page-sitemap.xml` содержит **только главную страницу** (1 страница). Остальные 17 страниц (услуги, прайс, калькулятор, контакты, о нас, блог, работы, запись, подписка, тест) в sitemap отсутствуют. Это настройка Yoast SEO (вероятно, страницы исключены из sitemap).
- `post-sitemap.xml` содержит 10+ статей блога.

### Пересчитанный приоритет

**P1** — частичная проблема: страницы не в sitemap, статьи есть.  
Но не P0, так как sitemap не пустые при валидном запросе. Нужно настроить Yoast: включить все публичные страницы в sitemap.
